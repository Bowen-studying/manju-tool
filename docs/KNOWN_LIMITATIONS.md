# 实际使用限制与风险

本文档列出 0.6.0 在真实制作中仍然存在、但不适合由本次代码修改“假装解决”的边界。接收者在选型和上线前应逐项确认。

## 1. 第三方 API 并不完全标准化

工具兼容常见 OpenAI 风格的 LLM、图片、语音和视频接口，但不同供应商仍可能使用不同字段、鉴权头、图片上传格式、任务状态或轮询地址。当前已覆盖的契约见 `docs/API_COMPATIBILITY.md`。

- 图片供应商若不支持 `/images/edits`，本地参考图可能只能走供应商自定义协议。
- 视频供应商若拒绝 data URL，本地参考图必须先放到可访问且受控的 URL。
- 切换供应商后建议先用一个低成本镜头做连通性测试，再运行长篇批量任务。

## 2. 生成质量仍受模型随机性影响

多阶段生成、JSON 修复和结构校验降低了格式错误，但不能保证模型每次都给出逻辑正确、角色一致或可拍摄的分镜。失败阶段会保留在 `stages/run_<fingerprint>/`，可修正输入或配置后续跑。重要项目仍需人工审核人物设定、连续性、台词、时长和镜头可执行性。

## 3. “素材流水线”不等于“自动成片”

pipeline 可以生成剧本、分镜、图片、配音、视频提示词和可选视频片段，但不负责最终时间线剪辑、字幕样式、口型同步、混音、转场、调色和成片质检。这些工作仍需剪辑软件或后续专用流水线完成。

## 4. 成本、限流和运行时间不可预测

LLM、生图、TTS 和视频服务可能计费并触发限流。`--render-videos` 默认关闭是为了避免误调用高成本接口。长篇项目即使可续跑，也可能运行数十分钟到数小时；目前各镜头主要按顺序处理，没有跨供应商的并发调度器或全局预算上限。

## 5. 角色与画面一致性不是强保证

场景级参考图、风格锚点和内容缓存能提高一致性，但最终效果取决于模型是否支持参考图、参考强度、种子和角色锁定能力。服装、脸型、道具和空间关系仍可能漂移，必要时应使用供应商的角色 LoRA、固定 seed 或人工挑图。

## 6. 语音后端依赖网络与供应商能力

默认 Edge TTS 需要网络，服务可用性和音色列表可能变化。自定义 TTS API 的情绪、语速、音量和音色字段也不一定全部生效。工具会稳定分配角色音色，但不提供声纹克隆授权、发音词典或专业级响度归一化。

## 7. 缓存和阶段文件会持续占用磁盘

时间戳输出目录、防覆盖版本、阶段快照、图片、音频、视频和 `.manju.json` 元数据都会保留。这样做有利于审计和续跑，但长项目需要自行制定归档与清理策略。清理前应保留当前 `storyboard.json` 和对应素材，避免破坏恢复链路。

## 8. PDF 排版存在后端差异

正式依赖和默认 PDF 后端可在无额外系统动态库的环境运行；默认后端不可用时才尝试兼容后端。两种后端的分页、字体和表格换行可能略有不同，超长提示词仍应人工检查 PDF；Excel 和 Markdown 保留完整文本，适合作为原始审查来源。

## 9. 隐私、版权和内容合规由使用者负责

脚本、小说、提示词和参考图会发送给所配置的第三方服务。工具不会替供应商提供本地加密、数据保留控制、版权确认、肖像授权或内容审核。敏感项目应使用受控服务，并先核对日志、训练使用和数据删除政策。

## 10. 依赖目前没有精确锁定

错误且过时的旧 `uv.lock` 已删除，当前以 `pyproject.toml` 和 `requirements.txt` 的版本下限为准。这避免安装错误依赖，但未来最新版本仍可能产生兼容变化。正式部署建议在目标系统完成一次验证后生成自己的锁文件或固定版本，并纳入发布流程。

## 11. 本次没有调用真实计费 API

自动测试覆盖了请求格式、重试、同步/异步响应、缓存、恢复和状态回写，但使用 mock 隔离第三方服务，没有消耗真实 API 额度。上线前仍需使用接收方自己的服务商和低额度账号做一次端到端冒烟测试。
